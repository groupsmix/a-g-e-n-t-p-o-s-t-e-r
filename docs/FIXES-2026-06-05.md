## Fixes applied

### Bug 1 â€” `/api/stats` graceful degrade

- `apps/dashboard/src/app/api/stats/route.ts`: instead of throwing, returns
  `{ configured: false, error: "..." }` with status `200`. The dashboard
  detects this and renders a setup screen that tells the operator exactly
  what to do, both locally and on Vercel.
- `apps/dashboard/src/app/page.tsx`: new "Setup required" branch.
- `apps/dashboard/.env.local.example`: new file documenting every variable
  the dashboard reads.

### Bug 2 â€” `/api/trigger` default repo + diagnostics

- `apps/dashboard/src/app/api/trigger/route.ts`:
  - Default repo is now `groupsmix/a-g-e-n-t-p-o-s-t-e-r`.
  - New `GITHUB_REF` env var so non-`main` workflows can be dispatched.
  - GitHub error responses are now surfaced with specific, actionable
    messages (404 â†’ "workflow file missing", 401/403 â†’ "token rejected").
  - Returns `503` instead of `500` when `GITHUB_TOKEN` is missing, with
    instructions for both local + Vercel setup.

### Bug 3 â€” Visible API misconfig warning

- `apps/nexus/apps/web/src/lib/api.ts`:
  - Exports `API_BASE_IS_FALLBACK` and `isApiMisconfigured()`.
  - Logs a loud `console.error` on module load when running in a non-
    localhost browser with the fallback API base.
- `apps/nexus/apps/web/src/components/shared/ApiMisconfigBanner.tsx`: new
  banner component shown on every page when the misconfig is detected.
- `apps/nexus/apps/web/src/components/shell/AppShell.tsx`: renders the
  banner at the top of `<main>`.

This is intentionally non-breaking: the localhost fallback still works in
local dev, but the moment a real production browser hits it, the operator
sees a clear, persistent warning instead of a silently empty dashboard.

### Bug 4 â€” Stub pages for the 3 truly missing sidebar links

- `apps/nexus/apps/web/src/app/content/page.tsx`
- `apps/nexus/apps/web/src/app/affiliate-marketing/page.tsx`
- `apps/nexus/apps/web/src/app/ecommerce-retail/page.tsx`

Each stub is honest about what is and isn't implemented, and links to the
closest live features so users have somewhere useful to land. No more 404s
from the default sidebar.

### Bug 5 â€” Documented in the stub pages

The two routes without backends (`/affiliate-marketing`, `/ecommerce-retail`)
include a "For maintainers" callout that names the exact file path where
the missing route module needs to live.

### Bug 6 â€” Dashboard deploy config

- `apps/dashboard/vercel.json`: monorepo-aware Vercel config.
  - `buildCommand` installs from the workspace root so pnpm catches all
    deps, then builds just the dashboard.
  - `ignoreCommand` skips Vercel builds when nothing under
    `apps/dashboard/`, `packages/`, or `pnpm-lock.yaml` changed â€” saves
    build minutes when iterating on NEXUS.

### Root env example refresh

`.env.example` at the repo root now lists `GITHUB_TOKEN`, `GITHUB_REPO`,
`GITHUB_REF`, and `NEXT_PUBLIC_API_URL`, with comments explaining what each
one does.

## What this pass does NOT do

I did not pick between **Path A** and **Path B**. That's a strategic call
that needs your input.

- **Path A â€” promote NEXUS web to "the" dashboard.** Keep
  `apps/nexus/apps/web`. Wire `packages/publishers`, `packages/generators`,
  `packages/agents` into the NEXUS API services layer. Delete or archive
  `apps/dashboard`. Lowest effort, keeps the 48 working pages.
- **Path B â€” promote `apps/dashboard` to "the" dashboard.** Migrate NEXUS
  off Cloudflare D1 onto Supabase using
  `packages/core/src/db/schema.sql`. Port the 48 NEXUS pages over. Highest
  effort, single unified system.

Until you decide, both dashboards co-exist. The bugs above stop them from
being silently broken in either direction.
