### Audit results

#### ‚úÖ Sidebar ‚Üí pages mapping

Every link in `Sidebar.tsx` resolves to a real Next.js route. Scan
performed by extracting every `to:` field from `defaultSections` and
matching against `apps/nexus/apps/web/src/app/**/page.tsx`.

Two short pages (`/trends`, `/winners`) are intentional 5-line redirects
to `/intelligence`. Not dead ‚Äî just compact navigation glue.

#### ‚úÖ Client ‚Üí Worker route alignment

Every top-level path the web client calls (`/api/X/...`) maps to a
mounted Hono route on `nexus-api` (`api.route('/X', ...)`).

No broken contracts. No 404s from the dashboard to the API for any wired
feature.

#### ‚öÝÔ∏è Unused client API methods (34, kept intact)

`lib/api.ts` defines 34 `api.X()` methods that no page currently calls.
Examples: `createGumroadProduct`, `nicheFactory`, `updateAIModel`,
`recordABEvent`.

**Decision: kept.** Each is a 1-line wrapper around `fetch`. They cost
nothing to keep and serve as a complete index of "what the Worker can do"
when wiring future UI. Deleting them would lose that index.

If any of these grow stale (e.g. the corresponding Worker route is
renamed), typecheck will catch it at build time.

#### ‚öÝÔ∏è Worker routes never called from the web client (6, kept intact)

Top-level mounts on `nexus-api` that no client page calls:

- `/api/signals`
- `/api/events`
- `/api/portfolio`
- `/api/offers`
- `/api/ventures`
- `/api/tracked-links`

**Decision: kept.** These are real route handlers using production-quality
service layers (`signal-normalizer`, `attribution-service`,
`factory-dispatcher`, etc.) ‚Äî see the Worker audit below for why they
stay even when the UI hasn't wired them yet.

#### ‚úÖ No orphan components remaining

After this pass: every component under `src/components/` is imported
somewhere in the tree.

#### ‚úÖ No broken imports

`pnpm --filter @nexus/web typecheck` passes clean.

#### ‚úÖ No TODO/FIXME debt

Zero occurrences across `apps/nexus/apps/web/src/`.

### How to re-run the dashboard audit

```bash
# From repo root, after `pnpm install`:

# 1. Sidebar links resolve to pages?
WEB=apps/nexus/apps/web/src
grep -oE "to: ['\"]/[^'\"]+" $WEB/components/shell/Sidebar.tsx \
  | sed "s/to: ['\"]//" | sort -u > /tmp/sidebar.txt
find $WEB/app -name page.tsx \
  | sed "s|$WEB/app||;s|/page.tsx||" | sort -u > /tmp/pages.txt
comm -23 /tmp/sidebar.txt /tmp/pages.txt   # links with no page

# 2. Client API calls covered by Worker mounts?
grep -oE "apiFetch[^(]*\(['\"]/[^'\"]+" $WEB/lib/api.ts \
  | grep -oE "/api/[a-z][a-z0-9-]*" | sort -u > /tmp/client-api.txt
grep -oE "api\.route\(['\"][^'\"]+" \
  apps/nexus/apps/nexus-api/src/index.ts \
  | sed "s/api.route(['\"]//;s|^|/api|" | sort -u > /tmp/worker-mounts.txt

# 3. Typecheck the whole web app
pnpm --filter @nexus/web typecheck
```

### Audit results

#### ‚úÖ Route files vs mounts (1:1)

48 handler files in `apps/nexus-api/src/routes/`, 48 imports in
`index.ts`, 49 `api.route()` mounts (one alias: `/categories` reuses
`domainRoutes`, `/niches` reuses `scoringRoutes`). The only `.ts` file
under `routes/` not imported is `auth.test.ts` ‚Äî that's a Vitest test
file, not a handler.

No route file is defined but not mounted. No mount references a
non-existent handler.

#### ‚úÖ Cron handler (daily 7am UTC) wires every imported helper

`wrangler.toml` has one cron trigger: `0 7 * * *`. The `scheduled`
function in `index.ts` calls:

- `sweepStaleRuns(env)` ‚Üí `services/sweep.ts` ‚úÖ
- `runTrendRadar(env)` ‚Üí local function in `index.ts` ‚úÖ
- `runDueSchedules(env, ctx)` ‚Üí `routes/schedules.ts` ‚úÖ
- `runAutopilot(env, ctx)` ‚Üí `routes/autopilot.ts` ‚úÖ
- `runMarketing(env, ctx)` ‚Üí `routes/marketing.ts` ‚úÖ
- `backfillDeliverables(env)` ‚Üí `services/deliverable.ts` ‚úÖ
- `sendDailyDigest(env)` ‚Üí `services/digest.ts` ‚úÖ
- `runLearningSync(env)` ‚Üí `routes/learning.ts` ‚úÖ
- Dynamic: drains job queue via `services/job-queue` + `services/agents` ‚úÖ

All 9 cron helpers resolve to real, exported functions. No dead cron
references.

#### ‚öÝÔ∏è Orphan cluster (deleted)

The deleted files referenced each other and `portfolio-integration.test.ts`
in a closed graph. None were:

1. Imported by any mounted route handler.
2. Called by the cron `scheduled` function.
3. Called via `env.SELF` or `env.AI_WORKER` self-binding.
4. Imported by the `nexus-ai` Worker.
5. Referenced from the web client.

Full verification methodology: per file, list every top-level export
(`function`, `const`, `class`, `interface`, `type`); for each export
name, `grep -rln "\b<name>\b" apps/nexus-api/src/` excluding the file
itself; if every consumer is also in the dead cluster, the file is dead.

#### ‚öÝÔ∏è Partial-dead exports (kept, flagged as follow-up)

Two production files have some live exports and some dead exports:

- `services/capital-allocator.ts`
  - ‚úÖ `allocateVenture` ‚Äî used by `routes/events.ts` (dynamic import) and `services/factory/factory-dispatcher.ts`
  - ‚ùå `runAllocatorForOpportunity` ‚Äî no callers
  - ‚ùå `runGlobalAllocator` ‚Äî no callers
- `services/deletion.ts`
  - ‚úÖ `deleteProduct`, `deleteDomain`, `deleteAsset` ‚Äî used by `storage/d1.ts`
  - ‚ùå `deletePlatform`, `deleteSocialChannel` ‚Äî no callers

**Decision: kept.** Trimming exports inside a still-imported file is a
larger surface change and the dead functions don't pull in extra
runtime cost. Worth a future small PR if you want strict cleanliness.

#### ‚öÝÔ∏è Duplicate CRUD in `routes/portfolio.ts` (kept, flagged as follow-up)

`routes/portfolio.ts` (947 lines, 23 handlers) and the standalone routes
(`signals.ts`, `ventures.ts`, `offers.ts`, `events.ts`, `tracked-links.ts`)
were created in the same commit and overlap substantially:

| `portfolio.ts` path | Equivalent standalone path | Standalone is more capable? |
|---|---|---|
| `GET/POST/PUT /api/portfolio/signals` | `/api/signals` | ‚úÖ has normalization, scoring, batch, promote |
| `GET/POST/PUT /api/portfolio/ventures` | `/api/ventures` | ‚úÖ has `/build` action via factory |
| `GET/POST/PUT /api/portfolio/offers` | `/api/offers` | ‚úÖ has approve/pause/kill/clone |
| `GET/POST /api/portfolio/tracked-links` | `/api/tracked-links` | ‚úÖ uses `attribution-service`, has `/click` |
| `GET/POST /api/portfolio/economic-events` | `/api/events` | ‚úÖ has `/ingest`, `/summary` aggregation |

`routes/portfolio.ts` ALSO has 11 unique handlers not covered elsewhere
(`/asset-library`, `/allocator-actions`, `/scoreboard`, `/allocation`,
`/cashflow`) backed by real D1 tables (`asset_library`, `allocator_actions`
exist in `migrations/021_portfolio_spine.sql`).

**Decision: kept this PR.** Trimming the duplicate handlers is a URL
contract change (even though nothing currently consumes
`/api/portfolio/*`). Worth a focused follow-up PR that:

1. Confirms the standalone routes are the keepers.
2. Removes lines 20-509 of `routes/portfolio.ts` (the duplicates).
3. Leaves the unique portfolio handlers intact.

### How to re-run the Worker audit

```bash
# From repo root, after `pnpm install`:

API=apps/nexus/apps/nexus-api

# 1. Every routes/*.ts is imported by index.ts?
ls $API/src/routes/*.ts | xargs -n1 basename | sed 's/.ts$//' | sort > /tmp/route-files.txt
grep -oE "from './routes/[a-z-]+" $API/src/index.ts | sed "s|from './routes/||" | sort -u > /tmp/route-imports.txt
comm -23 /tmp/route-files.txt /tmp/route-imports.txt  # files not imported (auth.test is OK ‚Äî vitest)

# 2. Every import is mounted via api.route()?
grep -oE "api\.route\(['\"]/[a-z-]+" $API/src/index.ts | sed "s|api.route(['\"]/||" | sort -u > /tmp/mounted.txt

# 3. Per-service usage scan (replace SERVICE):
SERVICE=capital-allocator
grep -oE "^export (async )?(function|const|class|interface|type) [A-Za-z_]+" \
  $API/src/services/$SERVICE.ts | awk '{print $NF}' | while read name; do
  count=$(grep -rln "\b$name\b" $API/src/ | grep -v "/$SERVICE.ts$" | wc -l)
  printf "%-40s used in %d files\n" "$name" "$count"
done

# 4. Typecheck
pnpm --filter @nexus/nexus-api typecheck
```
